# List Snapshot endpoint and cache key

The public API will expose `GET /v1/list-snapshots` with a `sharedListLink` query parameter, returning a List Snapshot rather than creating stored list resources. Public delivery code calls Google Maps List Acquisition by Google Maps Shared List Link only; Google Maps List ID remains an internal acquisition identity used for upstream requests and cache keys. Acquisition owns typed domain errors such as unsupported links, missing Google Maps List IDs, unavailable lists, upstream failures, and parse failures; delivery code maps those errors to HTTP status codes and response bodies. Acquisition may own plain Zod schemas for its domain data, but it must not depend on Hono or OpenAPI-specific schema helpers. Caching has two layers: Link Resolution caches the mapping from Google Maps Shared List Link to Google Maps List ID, and List Snapshots are cached for one hour by Google Maps List ID only. Google Maps locale inputs are fixed server-side configuration for now so public callers cannot create multiple snapshot variants or receive surprising first-writer locale behavior. Cache state is communicated through HTTP caching semantics rather than fields in the JSON response: normal responses use `Cache-Control: public, max-age=3600` and an `ETag`, and conditional requests may return `304 Not Modified`. Clear Google Maps unavailable/private/deleted responses are cached as negative entries for five minutes; transient upstream, network, and parser failures are not cached. Link Resolution Invalidation is handled by a secret-protected admin route; it is a local SQLite-only operation that deletes the Link Resolution and, when the cached Google Maps List ID is known, the downstream List Snapshot without making live Google requests.
